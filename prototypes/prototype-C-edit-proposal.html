AGF Spec v3.0

Pending change last edited 4 minutes ago iCloud · AGRC

This document defines the Agent Graph Framework (AGF) v3.0. It builds on v2.0 and introduces clarifications, refinements, and new capabilities to the runtime contract.

1. Workflows

In AGF v3.0, a flow is a directed graph of steps executed by an agent runtime. a workflow is a directed graph of steps executed by an agent runtime. Each step represents a deterministic unit of work with defined inputs, outputs, and execution semantics. Steps MAY invoke tools, call sub-workflows, or evaluate conditions.

Workflows MAY declare explicit version constraints on referenced components to ensure reproducibility across environments. Constraints are evaluated at compile time and resolved at runtime against the active component registry.

All workflows must be valid according to the schema defined in section 7. Invalid workflows are rejected at the compile boundary and never reach the runtime.

2. Step semantics

A step is the atomic unit of execution. Steps are typed: each step declares its input type, output type, and effect class. The runtime enforces type compatibility at every edge in the graph.

type Step = {
  id: UUID,
  kind: "tool" | "sub" | "cond",
  effect: "pure" | "read" | "write",
};

Effect classes drive the runtime's scheduling and rollback behavior. Pure steps may be re-executed freely; write steps are guarded by the journal described in section 5.

3. Execution model

The runtime executes a workflow by traversing its directed graph in topological order. Parallelism is permitted between any two steps with no transitive dependency. The runtime is responsible for ensuring that the observable result is identical to a serial execution in dependency order.
